# Galaxea 桌面异常场景 1-5

本文档记录当前实验最先使用的五个异常场景。它们围绕桌面任务 `抓取 apple` 以及 `将 apple 放到 box 上` 展开，并与当前的 field atomic 技能、经验库和异常分类保持一致。

## 共同设置

- 场景物体：`apple`、`red box`、`green box`
- 任务目标：默认是抓取 `apple`；放置类实验扩展为将 `apple` 放到 `red box` 或 `green box` 上
- 视觉输入：任务开始先采集 RGB-D，再做物体识别
- 约束：不引入旧 G3/G4 组合技能，只使用当前 Galaxea 原子技能
- 记录项：`scenario_id`、`target_class`、每步技能结果、`final_error`、`object_lift_world`、是否恢复成功

## 场景 1：错误目标选择

异常类型：

- `perception_miss`

场景说明：

- 桌面上同时放着 `apple`、`red box`、`green box`
- 初始计划故意把识别目标写错，例如把 `apple` 误写成 `red box`

实验目的：

- 验证模型是否能发现目标选错
- 验证恢复时能否重新识别并切回正确目标

推荐恢复方式：

1. 重新拍摄 RGB-D
2. 重新识别 `apple`
3. 用新的目标位置重新规划预抓取和接近
4. 再继续抓取和提升

典型判据：

- 是否最终抓到 `apple`
- 是否出现错误目标沿用
- 是否能成功重算位置并继续执行

## 场景 2：识别后目标位置失效

异常类型：

- `object_not_lifted` 的前置版本
- 也可视为位置过期类异常

场景说明：

- 先识别 `apple`
- 然后在真正执行 `move_to_pregrasp` 前，人为移动 `apple`
- 结果是旧的 `object_position.json` 已经过期

实验目的：

- 验证系统是否会继续相信旧位置
- 验证异常后是否会重新识别并刷新目标位置

推荐恢复方式：

1. 重新拍摄 RGB-D
2. 重新识别 `apple`
3. 重建目标位置
4. 重新规划并再次接近

典型判据：

- `move_to_pregrasp` 的误差是否明显下降
- 重识别后的目标是否接近真实物体
- 是否成功进入后续抓取

## 场景 3：抓取失败或滑脱

异常类型：

- `grasp_miss`
- `grasp_slip`

场景说明：

- 目标 `apple` 较小或接触面不稳定
- `close_gripper` 看似成功，但 `lift` 时物体没有稳定跟住

实验目的：

- 验证抓取点不准、闭合不足或夹持不稳时的恢复
- 验证系统是否会通过重新识别、调整接近参数再次抓取

推荐恢复方式：

1. 先打开夹爪
2. 重新识别 `apple`
3. 重新规划预抓取和接近
4. 适当调整接近偏移后再次抓取

典型判据：

- `object_lift_world` 是否足够大
- 是否出现掉落或接触丢失
- 是否在第二次尝试中恢复成功

## 场景 4：可达性不足

异常类型：

- `actuation_limit`
- 或接近阶段误差持续偏大

场景说明：

- `apple` 放在机械臂当前姿态较难够到的位置
- 直接抓取时 `move_to_pregrasp` 或 `approach_object` 误差过大

实验目的：

- 验证大模型是否会主动调用底盘相对移动
- 验证移动后是否重新感知并再次抓取

推荐恢复方式：

1. 先执行底盘相对移动
2. 重新拍摄和识别
3. 重新规划接近轨迹
4. 再执行抓取

典型判据：

- 底盘移动后目标是否进入可达范围
- 接近阶段误差是否下降
- 最终是否完成抓取和提升

## 场景 5：放置位不稳定或被占用

异常类型：

- `place_occupied`
- `place_error`
- 如果 apple 在放置过程中掉落，也记录为 `grasp_slip` 或 `object_not_lifted`

场景说明：

- 任务是抓起 `apple`，并把它放到 `red box` 上方
- `red box` 顶面可能已被占用，或者放置点太靠边
- 也可以故意给出不稳定的 `place_offset_x/y/z`，让 apple 落在 box 边缘或支撑面积不足的位置

实验目的：

- 验证系统是否能识别放置位不稳定或被占用
- 验证异常处理时是否能重新选择放置位
- 如果 apple 掉落，验证系统是否会先重新抓取 apple，再重新执行放置

触发方式：

1. 正常识别并抓起 `apple`
2. 识别 `red box` 作为放置目标
3. 在 `transport_to_detected_target` 中设置小范围 `place_offset_x/y`，让放置点偏向 box 边缘
4. 使用 `lower_held_object` 将 apple 向下靠近 box 顶面
5. 打开夹爪后观察 apple 是否滑落、掉落或离开目标区域

恢复逻辑：

如果 apple 仍然在夹爪中：

1. 重新识别 `red box`
2. 改用更稳定的放置偏移
3. 必要时改用 `green box`
4. 再执行运输、下降和松爪

如果 apple 已经掉落：

1. `open_gripper`
2. 重新拍摄 RGB-D
3. 重新识别 `apple`
4. 重新规划预抓取和接近
5. 再次抓取并提升 `apple`
6. 重新识别 `red box` 或 `green box`
7. 选择新的稳定放置位
8. 再执行运输、下降和松爪

典型判据：

- apple 放置后是否留在目标 box 顶面
- apple 最终位置是否落入目标区域
- 是否出现明显滑落、翻倒或掉落
- `place_error` 是否降低
- 如果发生掉落，第二次是否能重新抓起 apple 并完成放置

## 建议顺序

优先顺序建议如下：

1. 错误目标选择
2. 识别后目标位置失效
3. 抓取失败或滑脱
4. 可达性不足
5. 放置位不稳定或被占用

这五个场景覆盖了当前最核心的感知错误、位置过期、抓取不稳、可达性不足和放置失败问题，适合作为异常处理闭环的第一批实验。
