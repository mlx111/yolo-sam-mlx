# 双源经验库设计与使用说明

本文档说明 `experiment-sim-wrapper1` 中当前已经实现的 Sim-Real 双源经验库：它的数据结构怎么设计，仿真经验和真实/伪真机经验怎么进入库，以及在线策略如何检索、评分和使用这些经验。

## 1. 设计目标

双源经验库不是单纯把历史恢复方案存成日志，而是把一次异常恢复 episode 组织成可检索、可验证、可比较的 `memory_v3_plus` 经验条目。当前实现重点解决三件事：

1. 保留仿真经验的可重复性：仿真实验可以稳定生成条件、异常注入、感知、恢复方案、执行反馈和关键帧。
2. 接入真实/伪真机经验：真实执行或从实验结果转成的伪真机 episode 会被归一化成同一个 MemoryV3+ schema。
3. 在在线策略中区分“支持证据”和“风险证据”：成功、真实验证、低 gap 的经验提高方案可信度；失败、高 sim-real gap、高 critic 风险的经验会降低方案分数，必要时触发重写。

当前实现的核心文件：

| 文件 | 作用 |
| --- | --- |
| `memory/v3.py` | MemoryV3+ schema、读写、检索、基础评分、双源字段 |
| `memory/dual_source.py` | sim-real 配对和 gap 计算 |
| `real_memory/ingest.py` | 真实/伪真机 episode 导入并归一化为 MemoryV3+ |
| `memory/scoring.py` | gap/risk 感知检索加权和 candidate plan 打分 |
| `memory/gating.py` | episode 写入价值评估信号 |
| `memory/critic.py` | rule/LLM critic 结果归一化 |
| `experiment_method_runner.py` | 在线策略方法、经验读写、双源检索、重写触发 |
| `tools/run_mvp_dual_source_pipeline.py` | 离线 MVP 流水线 |
| `tools/import_real_episode.py` | 导入真实/伪真机 episode |
| `tools/build_sim_real_pairs.py` | 对已有 memory 建立 pair/gap |

## 2. MemoryV3+ 的核心结构

所有仿真和真实经验最终都保存成 `memory_v3_plus` 条目，即 `memory/v3.py::MemoryV3Entry`。一条经验包含以下信息：

| 字段 | 含义 |
| --- | --- |
| `condition_id`, `scenario_id` | 异常条件和场景，例如 `U4-3` / `U4` |
| `available_actions`, `skill_sequence` | 当前场景允许动作，以及实际执行/计划的技能序列 |
| `result`, `status` | 恢复是否成功、任务是否成功、失败原因 |
| `source`, `validation_status`, `memory_partition` | 经验来源和分区，例如 simulation、real、failed_memory、real_memory |
| `anomaly`, `scene`, `task` | 异常、场景、任务阶段信息 |
| `perception`, `reconstruction_artifacts` | 感知结果和沙盒/虚拟场景重建信息 |
| `recovery_plan`, `execution_feedback` | 恢复方案和执行反馈 |
| `key_slices`, `keyframes` | episode 关键时间片和关键帧 |
| `retrieval_key`, `anomaly_state` | 用于结构化检索的摘要键 |
| `failure_taxonomy` | 失败分类、失败阶段、失败原因 |
| `memory_gate` | 写入价值评估结果 |
| `sim_real_pair`, `sim_real_gap` | 双源配对和 sim-real gap |
| `sandbox_calibration` | 根据 gap 估计的沙盒校准信号 |
| `critic_result` | rule/LLM critic 合并后的风险结果 |
| `real_episode_ref` | 真实 episode 的外部引用，例如原始 episode id、视频目录、日志路径 |
| `memory_tags` | 经验角色标签，例如 `real_success_prior`、`real_failure_case`、`sim_real_gap_memory` |

分区由 `memory/v3.py::infer_memory_partition()` 自动推断：

| 条件 | 分区 |
| --- | --- |
| `validation_status` 为 `failed` / `failure`，或恢复失败 | `failed_memory` |
| `validation_status` 为 `real_executed` / `real_validated` | `real_memory` |
| `validation_status` 为 `simulation_validated` | `validated_memory` |
| 其他仿真成功经验 | `simulation_memory` |

`retrieval_key` 是在线检索的关键结构，由 `build_retrieval_key()` 生成，包含：

- `condition_id`, `scenario_id`, `task_stage`
- `plan_signature`
- `contact_pattern`
- `lift_success`, `failure_type`
- `gap_type`, `gap_uncertainty_bucket`
- `critic_status`, `critic_flags`
- `memory_type`, `memory_role`
- `real_validated`, `pair_status`

## 3. 仿真经验怎么获取

仿真经验主要由 `experiment_method_runner.py::ExperimentMethodRunner.save_experience()` 写入。

典型流程是：

1. `run_experiment_batch.py` 为每个 condition/method/trial 启动 `experiment_method_runner.py` 子进程。
2. `ExperimentMethodRunner` 继承 `run_experiment_v4.py::ExperimentV4`，执行异常注入、感知检测、恢复规划、恢复执行、结果评估。
3. `ExperimentV4._build_experience_entry()` 将一次 trial 的指标打包成 MemoryV3+ entry。
4. `ExperimentMethodRunner.save_experience()` 决定是否写入当前经验库，并补充 critic、失败分类、视觉索引等信息。

仿真经验中会记录：

- 条件信息：`condition_id`、`scenario_id`、`failure_family`、`condition_injection`
- 感知信息：`perceived_position`、mask 数量、检测置信度、关键帧
- 恢复信息：LLM/技能计划、实际执行步骤、`executed_plan_source`
- 成败证据：`recovery_success_criteria`、`task_success_criteria`、接触状态、物体高度变化、pinch 距离
- 失败 critic：确定性 rule critic 会对所有失败 entry 运行；hierarchical 失败经验还会尝试 LLM critic
- `critic_result`：由 `memory/critic.py::build_critic_result()` 合并 rule/LLM critic，形成 `overall_status`、`critic_risk_score` 和各类 rule flags

写入策略由 `--experience-save-mode` 控制：

| 模式 | 行为 |
| --- | --- |
| `all` | 成功和失败都写入 |
| `success_only` | 只保存成功经验，失败会跳过 |
| `none` | 不保存当前 trial 经验 |

如果写入目录旁边有视觉索引，关键帧会进入 `visual_index/`，后续可参与视觉检索。

## 4. 真实/伪真机经验怎么获取

真实/伪真机经验由两层代码处理：

1. `real_memory/collector.py` 负责从真机产物目录收集 episode 元数据、传感器摘要和外部日志引用。
2. `real_memory/ingest.py` 负责把标准 episode JSON 归一化为 MemoryV3+ entry。

当前实现支持三类输入：

1. 手工整理的真实 episode JSON。
2. 已有实验结果 `result.json` 转成的伪真机 episode。
3. 真机运行后落盘的 episode 目录，通过 `FileEpisodeCollector` 自动组装标准 JSON。

最小必需字段在 `real_memory/schema.py` 中定义：

- `condition_id`
- `scenario_id`
- `executed_recovery_steps`
- `recovery_success`

导入流程：

1. `normalize_real_episode()` 从原始 JSON 中提取 condition、任务阶段、执行步骤、成败、关键帧和外部引用。
2. `real_episode_to_memory_entry()` 将其转成 MemoryV3+ entry。
3. entry 的 `source` 固定为 `real`，`validation_status` 为 `real_executed`。
4. `memory_gate` 会通过 `compute_memory_gate()` 记录这条经验为什么值得保存。
5. `real_episode_ref` 保存外部追溯信息，例如原始 episode id、视频目录、keyframe 目录、机器人日志路径和传感器模态。
6. `memory_tags.memory_role` 根据成败标记为 `real_success_prior` 或 `real_failure_case`。

可直接使用工具导入：

```bash
python -B tools/import_real_episode.py \
  --input path/to/real_or_pseudo_real_result.json \
  --memory-out results/real_memory_snapshot.json \
  --report results/real_memory_import_report.json
```

如果真机系统已经把一次 episode 的文件落到目录中，可以先采集再导入：

```bash
python -B tools/collect_real_episode.py \
  --episode-dir path/to/real_episode_dir \
  --episode-out results/collected_real_episode.json \
  --memory-out results/real_memory_snapshot.json \
  --report results/real_episode_collect_report.json
```

`FileEpisodeCollector` 默认识别这些文件或目录：

| 文件/目录 | 含义 |
| --- | --- |
| `episode.json` / `real_episode.json` / `result.json` | episode 元数据、任务结果、执行动作序列 |
| `sensor_summary.json` / `sensors.json` | 可选的关节、末端、夹爪、接触、时间戳等压缩摘要 |
| `robot_log.jsonl` / `robot_log.json` | 控制器或技能执行原始日志；没有 `sensor_summary.json` 时会自动解析 |
| `episode.hdf5` / `episode.h5` | 高频时序数据引用；当前支持从常见 dataset 路径自动提取摘要 |
| `video/` / `videos/` / `rgb/` / `camera/` | 图像或视频目录 |
| `keyframes/` / `frames/` | 关键帧目录 |

真实机器人接入 ROS、ROS2 或厂商 SDK 时，不需要改 MemoryV3+ 结构；只需要实现一个新的 `RealEpisodeCollector` adapter，输出与 `FileEpisodeCollector.collect()` 相同的标准 JSON。

当前代码还提供 `real_memory/log_parsers.py` 和 `tools/parse_real_sensor_log.py`。前者支持从 JSONL、JSON samples 和 HDF5 自动生成 `sensor_summary`，默认取 episode 最后一帧作为最终机器人状态，并记录 `timestamps.start`、`timestamps.end` 和 `timestamps.num_samples`。字段别名兼容 `qpos`/`joint_positions`、`qvel`/`joint_velocities`、`ee_pose`/`end_effector_pose`、`gripper_width`/`pinch_distance`、`contact`/`contact_state` 等常见写法。rosbag 解析目前保留接口，待确定 ROS1/ROS2 topic 后实现。

当前 MVP 配置中，`configs/mvp_dual_source_gap_critic_u3_u4_v1.json` 和 `configs/mvp_dual_source_gap_critic_u4_3_v1.json` 使用历史实验 `result.json` 作为 `pseudo_real_inputs`。这些输入是“伪真机”来源，用于验证双源链路，不等同于已经接入真实机器人在线采集。

### 4.1 真机传感器数值怎么保存

真机经验需要记录具体传感器数值，但不应把完整高频时序全部塞进经验库 JSON。当前设计采用两层保存：

1. MemoryV3+ entry 保存压缩后的关键证据，用于检索、pair/gap、critic 和 candidate scoring。
2. 原始时序数据保存到外部 HDF5、rosbag、视频目录或机器人日志中，通过 `real_episode_ref.hdf5_path`、`real_episode_ref.video_dir`、`real_episode_ref.robot_log_path` 引用。

建议采集的真机/仿真共同字段如下：

| 类型 | 具体数值 | 在经验库中的用途 |
| --- | --- | --- |
| 关节状态 | `qpos` / joint positions、`qvel` / joint velocities | 判断执行偏差、关节限制、轨迹是否可复现 |
| 末端状态 | end-effector pose、gripper pose、pinch site pose | 与物体位置比较，判断是否到达抓取/放置区域 |
| 夹爪状态 | gripper command、gripper width、pinch distance、夹爪电流/力矩 | 计算 actuation gap、判断是否夹稳 |
| 接触/力觉 | contact flags、contact pair、force/torque、触觉或夹爪力估计 | 计算 contact gap、识别滑移/掉落/虚假接触 |
| 视觉输入 | RGB、depth、mask、bbox、keyframes | 支持 perception gap、视觉检索和人工复核 |
| 感知输出 | target class、2D bbox、mask area、3D perceived position、confidence | 区分真实物体状态和感知估计误差 |
| 执行结果 | observed object pose、object z after recovery、lift distance、drop/collision flags | 形成 recovery/task success 判据 |
| 时间信息 | step timestamp、episode start/end、动作持续时间 | 对齐 sim-real episode，定位异常发生阶段 |

这些值在仿真中通常可由 MuJoCo 直接获得：`qpos`、`qvel`、body/site pose、actuator ctrl、contact、RGB/depth render、物体高度和夹爪距离。真机中则由机器人控制器、夹爪驱动、相机、深度相机、力/触觉传感器或外部标定系统获得。真机如果没有直接物体真值位置，`observed_pos` 应来自视觉/标定/人工标注，不能当作无噪声真值。

进入长期经验库时，应优先保存这些摘要字段：

- `execution_feedback.observed_pos`
- `execution_feedback.contact_after_close`
- `execution_feedback.contact_after_lift`
- `execution_feedback.recovery_success_criteria`
- `perception.after_anomaly.object_pos`
- `perception.after_anomaly.confidence`
- `sensor_summary.joint_positions`
- `sensor_summary.joint_velocities`
- `sensor_summary.end_effector_pose`
- `sensor_summary.gripper_state`
- `sensor_summary.contact_state`
- `sensor_summary.raw_refs`
- `keyframes`
- `validation_evidence`
- `real_episode_ref`

完整传感器时序保存在外部引用中。这样既能让经验库参与在线检索和风险评估，又能在需要时回放原始 episode 做复核、重新标注或训练后续 critic/gap model。

当前代码已实现上述摘要字段的最小闭环：仿真运行会在 MemoryV3+ 中写入 `sensor_summary`，包括 MuJoCo 关节位置/速度、pinch site 位姿、夹爪 driver joint、夹爪距离、接触摘要和结果文件引用；真机/伪真机导入会透传输入 JSON 中的 `sensor_summary`，并补充 HDF5、视频、keyframe、机器人日志等外部引用。sim-real gap 计算会输出 `robot_state_gap`，用于记录关节位置误差和夹爪状态差异；当前这些机器人状态差异先作为解释性证据保存，尚未纳入总 gap score 的权重。

## 5. Sim-Real 配对和 Gap 计算

双源配对由 `memory/dual_source.py` 完成。

`pair_score(sim_entry, real_entry)` 判断两条经验是否可比较，主要依据：

| 信号 | 加分含义 |
| --- | --- |
| `scenario_id` 相同 | 必须同场景，否则分数为 0 |
| `condition_id` 相同 | 同一异常条件更可信 |
| `plan_signature` 相同 | 执行动作序列一致 |
| `contact_pattern` 相同 | 接触模式一致 |
| `task.stage` 相同 | 任务阶段一致 |

`pair_sim_real_experiences()` 会按 pair score 排序，为每条 real entry 选择一个最佳 sim pair，默认最低阈值是 `0.55`。

配对后，`compute_sim_real_gap()` 计算可解释 gap：

| gap 字段 | 说明 |
| --- | --- |
| `outcome_gap` | sim/real 成败是否一致，包含 `matched_success`、`matched_failure`、`sim_success_real_fail`、`sim_fail_real_success` |
| `pose_gap` | 执行后观测位置、恢复后 z 高度的差异 |
| `contact_gap` | sim/real 接触模式是否不一致 |
| `perception_gap` | 感知位置差异 |
| `actuation_gap` | pinch distance 等夹爪闭合差异 |
| `gap_score` | 综合 gap 分数 |
| `uncertainty` | 证据缺失和 gap 大小时的风险不确定性 |

其中 `pose_gap`、`contact_gap`、`perception_gap` 和 `actuation_gap` 都依赖上述传感器摘要：例如物体观测位置、感知位置、接触状态、pinch distance、夹爪宽度、关节/末端状态等。仿真和真机都能提供这些信号，但真机侧需要通过控制器日志、相机/深度相机、夹爪状态或外部标定系统采集，并通过 `real_episode_ref` 保留原始数据引用。

`apply_pair_and_gap()` 会返回 `sim_real_pair` 和 `sim_real_gap` 两个字典，写回 real entry。写回后，real entry 的 `memory_tags.memory_role` 通常会标记为 `sim_real_gap_memory`，并重新生成 `retrieval_key`。

可单独运行：

```bash
python -B tools/build_sim_real_pairs.py \
  --memory results/memory_v3_plus.json \
  --report results/gap_report.json \
  --write-memory results/sim_real_memory_snapshot.json
```

## 6. 离线 MVP 流水线

`tools/run_mvp_dual_source_pipeline.py` 将导入、配对、gap、检索和候选方案评分串起来。

当前流水线步骤：

1. 读取配置中的 `pseudo_real_inputs`。
2. 通过 `load_real_episode_entry()` 导入为 real memory。
3. 为每条 real entry 合成一个可比较的 sim entry，形成完整双源库。
4. 使用 `pair_sim_real_experiences()` 和 `apply_pair_and_gap()` 写入 pair/gap。
5. 保存：
   - `real_memory_snapshot.json`
   - `sim_real_memory_snapshot.json`
   - `gap_report.json`
   - `candidate_scoring_report.json`
6. 对配置中的 `conditions` 运行 gap/risk-aware retrieval 和 `score_candidate_plan()`。
7. 输出 Markdown 分析报告。

示例：

```bash
python -B tools/run_mvp_dual_source_pipeline.py \
  --config configs/mvp_dual_source_gap_critic_u4_3_v1.json
```

当前主要配置：

| 配置 | 条件 | 输出 |
| --- | --- | --- |
| `configs/mvp_dual_source_gap_critic_u3_u4_v1.json` | `U3-4`, `U4-2` | `results/mvp_dual_source_gap_critic_u3_u4_v1/` |
| `configs/mvp_dual_source_gap_critic_u4_3_v1.json` | `U4-3` | `results/mvp_dual_source_gap_critic_u4_3_v1/` |

## 7. 在线策略怎么使用双源经验

在线策略入口在 `experiment_method_runner.py`。

当前新增两个方法：

| method | condition | memory_policy | 含义 |
| --- | --- | --- | --- |
| `dual_source_gap_memory` | `sim_wrapper` | `dual_source_gap` | 使用双源 gap/risk aware 检索，但不强制因风险评分触发重写 |
| `dual_source_gap_critic` | `sim_wrapper` | `dual_source_gap_critic` | 使用双源检索，并在候选方案风险较高时触发一次重写 |

在线检索发生在 `_query_experiences_for_policy()`：

1. 使用当前 `scenario_id`、`condition_id`、allowed actions、`anomaly_state` 和 `retrieval_key` 调用 `MemoryV3Library.query()`。
2. 对双源策略，`top_k` 会放大到至少覆盖整个库，`diversity_lambda=0.0`，避免 MMR 过早丢掉风险证据。
3. 查询时开启：
   - `critic_prefilter=True`
   - `gap_aware=True`
   - `risk_aware=True`
   - 可选 `visual_index` 和当前 keyframes
4. `MemoryV3Library.query()` 会在基础结构化相似度外加入双源调整：
   - `real_validation_bonus` 表示真实/伪真机验证来源，`real_success_bonus` 会提高真实成功经验的排序
   - `paired_bonus` 给已经完成 sim-real 配对的经验少量稳定性加分
   - `sim_real_failure_penalty` 会显著降权 `sim_success_real_fail` 经验
   - `gap_score_penalty` 和 `gap_uncertainty_penalty` 会降低 gap 大或不确定性高的经验排序
   - `critic_penalty` 和 `failure_penalty` 会降低 critic 风险或失败经验排序
   - 这些解释会写入 `get_last_score_explanation()`
5. 双源策略把候选拆成两组：
   - support：不是 risk entry 的经验
   - risk：失败、gap uncertainty >= `0.8` 或 critic risk >= `0.5` 的经验
6. support 取 `POLICY_POSITIVE_TOP_K`，risk 取 `POLICY_FAILED_TOP_K`，再去重合并。
7. metrics 中记录：
   - `retrieved_memories`
   - `support_retrieved_memories`
   - `risk_retrieved_memories`
   - `dual_source_support_count`
   - `dual_source_risk_count`

## 8. 候选方案评分和重写触发

LLM 生成恢复步骤后，`ExperimentMethodRunner._after_llm_plan_generated()` 调用 `score_candidate_plan()`。

评分输入：

- `candidate_steps`
- 上一步检索到的 support/risk 双源经验

评分信号：

| 信号 | 作用 |
| --- | --- |
| `support_score` | 与成功经验动作序列重叠越高越好 |
| `real_success_prior` | 真实/伪真机成功 evidence 越多越好 |
| `failure_overlap_risk` | 与失败经验动作序列越重叠，风险越高 |
| `gap_uncertainty` | 对应经验 sim-real gap 越不确定，风险越高 |
| `critic_risk` | critic 认为有风险时降低分数 |

同时，`llm_handler.plan_recovery()` 会把 top-k 经验中的双源摘要写入 prompt。成功经验会显示 `来源`、`验证状态`、`Sim-Real Gap` 和可信提示；风险经验会显示 `sim_success_real_fail`、`gap_score`、`uncertainty` 和“仿真成功但真实/伪真机失败，不能直接照搬”的风险提示。这样 LLM 不只看到动作序列，还能看到该经验是否已经真实验证，以及是否存在 sim-real 迁移风险。

最终输出：

| decision | 含义 |
| --- | --- |
| `prefer` | 分数 >= 0.70，优先采用 |
| `allow` | 分数 >= 0.45，可执行 |
| `rewrite_recommended` | 分数 >= 0.25，建议重写 |
| `reject_recommended` | 分数 < 0.25，建议拒绝或重写 |

对于 `dual_source_gap_critic`：

1. 如果 candidate decision 是 `rewrite_recommended` 或 `reject_recommended`，`dual_source_rewrite_recommended=True`。
2. `_maybe_rewrite_blocked_plan()` 会检查 failed/risk blocker。
3. 如果没有普通 failed blocker，但双源评分建议重写，则通过 `_dual_source_risk_blocker_matches()` 从 risk evidence 构造 blocker。
4. 触发一次 `plan_recovery()` 重写。
5. 重写后再次记录 `rewritten_candidate_score` 和 `candidate_score_history`。

也就是说，`dual_source_gap_memory` 主要是“把双源证据放进检索和 prompt”；`dual_source_gap_critic` 进一步让双源风险评分影响执行前决策。

## 9. 写入价值 Gate 和 Critic 的作用

`memory/gating.py::compute_memory_gate()` 给 episode 计算写入价值，不直接决定是否保存，但会把价值信号写进 entry：

- `anomaly_score`
- `failure_score`
- `sim_real_gap_score`
- `recovery_utility_score`
- `surprise_score`
- `write_score`
- `write_decision`: `high_value`、`ltm_candidate`、`stm_only`、`raw_log`
- `trigger_events`

`memory/critic.py::build_critic_result()` 把确定性 rule critic 和 LLM critic 合成统一 `critic_result`：

- `overall_status`: `pass`、`warn`、`block`、`unknown`
- `critic_risk_score`
- `collision`、`joint`、`gripper_contact`、`end_effector_pose` 分项
- `rule_flags`
- `feedback_for_rewrite`

在线检索和 candidate scoring 都会使用 `critic_risk_score`。

## 11. 当前边界

1. 当前 MVP 中的“real”主要来自历史 `result.json` 导入的伪真机 episode；真实机器人在线采集链路还没有在此文档覆盖为已完成能力。
2. 双源 pair/gap 目前是一版可解释规则：按 condition、plan signature、contact、pose、outcome 等结构化字段计算，没有训练学习式 gap model。
3. `dual_source_gap_critic` 当前只触发一次重写，不做多轮搜索或多候选排序执行。
4. 视觉索引是可选增强；在线 smoke 不加载真实感知模型，也不验证 Grounded-SAM 视觉链路。
5. `memory_gate` 当前记录写入价值，真正是否保存仍由 runner 的 `experience_save_mode` 和调用方控制。
