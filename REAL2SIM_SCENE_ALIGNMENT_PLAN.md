# Real2Sim Scene Alignment Plan

## 1. 背景与目标

当前项目已经具备以下能力：

- 根据现实图像动态生成 MuJoCo 仿真场景
- 根据点云补全结果动态加载物体 mesh / STL
- 通过局部点云的相对位置关系，将物体放入仿真环境

当前仍未彻底解决的问题有两个：

1. 如何稳定确定仿真中相机的位姿，使 MuJoCo 中 `cam1/cam2` 看到的画面与真实相机拍摄画面一致
2. 如何让仿真环境中的物体位姿动态对应现实场景，而不是只保证位置近似正确或只保证单视角下的姿态看起来合理

本方案不修改现有代码，只给出一套基于当前仓库能力的可执行技术路线，并明确后续应如何组织 real2sim 对齐链路。


## 2. 当前项目链路与现状

当前主入口是：

- `zhenghe2_buquan.py`

当前与位姿最相关的模块分别是：

- `calibrate_runtime_pose_from_clouds.py`
- `pointcloud_v2.py`
- `object_pose_runtime.py`
- `dong2.py`
- `POSE_ALIGNMENT_RUNTIME.md`

当前链路大致如下：

1. 通过分割得到苹果、梨、机械臂等目标的 mask
2. 通过 `calibrate_runtime_pose_from_clouds.py` 构造 `runtime_pose_calibration.json`
3. 通过 `pointcloud_v2.py` 生成物体和机械臂的局部点云、背景点云
4. 通过 `pointcloud_v2.pos()` 得到苹果和梨的相对位置
5. 通过 `object_pose_runtime.py` 根据左相机原始点云和左图 mask 估计苹果和梨的姿态
6. 通过 `dong2.generate_scene(...)` 将相机位姿、物体位置、物体姿态写入 MuJoCo 场景

当前系统已经做到：

- 相机位置可以动态生成
- 相机旋转可以通过背景点云和 `cam1` 局部对齐进行修正
- 苹果、梨的位置已经恢复到合理量级
- 苹果、梨的姿态不再是手写固定值，而是基于左相机观测动态估计

但当前仍存在一个核心结构性问题：

- 相机位姿、物体位置、物体姿态来自三条不同逻辑链路
- 三者之间缺少统一的世界坐标系闭环验证
- 当前系统更接近“分模块拼装结果”，而不是“由统一观测约束共同解出的场景状态”


## 3. 当前问题的根因

### 3.1 相机位姿问题的根因

当前相机位姿虽然已经比早期版本合理很多，但本质上仍然是：

- 右相机主要由背景平面和世界系构造决定
- 左相机主要是在已有旋转基础上，通过机械臂和物体的 2D 锚点进一步修正

这个做法能得到“看起来比较对”的相机位姿，但还没有形成严格闭环，原因是：

- 相机位姿没有同时受到左右视角观测的统一约束
- 相机位姿没有和物体最终姿态一起联合优化
- 相机位姿和物体位置之间仍可能互相补偿误差

换句话说，当前 `cam1` 看起来合理，并不等于整个仿真场景与真实场景已经几何一致。

### 3.2 物体位姿问题的根因

当前物体位姿是拆开求的：

- 位置主要来自 `pointcloud_v2.pos()`
- 姿态主要来自 `object_pose_runtime.py`

当前 `object_pose_runtime.py` 的优点是：

- 已经不再依赖 `buquan` 的中间对齐坐标
- 已经回到原始左相机点云来估计姿态
- 已经支持多候选姿态 + mask 评分选择最佳结果

但它仍然存在两个限制：

1. 当前姿态估计本质上仍然是左视角主导
2. 当前姿态估计只解决旋转，平移仍然来自另一套逻辑

因此当前系统更像是：

- 用一套逻辑确定物体在世界中的位置
- 用另一套逻辑确定物体在世界中的旋转

这会导致一个典型问题：

- 即使左图投影看起来合理，换到右图或换到 MuJoCo 中的相机渲染后，仍可能出现轮廓、方位、遮挡关系不一致


## 4. 统一世界坐标系定义

要让仿真中相机看到的场景与现实相机拍摄的场景保持一致，必须先固定一个统一的世界坐标系。

本方案建议将 MuJoCo 世界系定义为：

- 原点：机械臂基座参考点
- `+Z`：竖直向上
- `+X`：桌面/背景主平面的水平参考方向
- `+Y = Z × X`

其中机械臂基座参考点沿用当前项目已经修正过的定义：

- `x/y` 使用机械臂点云包围盒中心
- `z` 使用机械臂点云最底端 `z_min`

这样做的原因是：

- 机械臂基座是整个场景中最稳定的参考对象
- 桌面 / 背景平面可以稳定提供竖直与水平几何约束
- 物体和相机都应相对这个世界系来表达，而不是彼此独立定义

这个定义应成为后续所有 real2sim 对齐的统一前提。


## 5. 相机位姿求解方案

### 5.1 总体原则

相机位姿不能只靠单个来源决定。

本方案建议将相机位姿视为一个需要被多种观测共同约束的量，而不是单独由某一张点云或某一组欧拉角推出。

相机位姿应同时满足：

- 与背景平面几何关系一致
- 与机械臂基座和机械臂主轴投影关系一致
- 与苹果、梨的左右图像位置一致
- 与左右相机之间的几何关系一致

### 5.2 相机位姿的推荐求解流程

#### 第一步：构造相机位姿初值

使用当前已有链路构造初值：

- 右相机初值：
  - 继续使用 `right1_background.ply` 的主平面法向构造世界系
  - 继续利用背景点云定义世界 `X/Y/Z`

- 左相机初值：
  - 继续基于当前 `delta_world @ current_left` 的方式，把左相机从旧世界映射到新世界
  - 继续使用 `cam1` 局部对齐器给出一个可用的修正起点

这里要强调：

- 当前 `runtime_pose_calibration.json` 应作为优化初值，不应直接视为最终真值

#### 第二步：加入机械臂几何约束

从左右相机中提取机械臂几何特征：

- 机械臂基座中心
- 机械臂主轴方向
- 机械臂上方参考点

利用这些几何特征作为 3D-2D 对齐锚点，对左右相机的外参施加约束。

机械臂几何约束的价值在于：

- 它比单纯物体中心更稳定
- 它比纯背景平面更接近操控任务真正关心的区域

#### 第三步：加入物体中心约束

从左右图的苹果、梨 mask 中提取：

- 中心点
- 主轴方向
- 轮廓

从左右相机局部点云中提取：

- 苹果中心
- 梨中心

要求在当前相机位姿下，物体投影中心与真实图像中的物体中心尽量一致。

#### 第四步：加入双目一致性约束

对同一个物体，左右相机都能得到局部点云。将两侧局部点云在当前外参下变换到统一世界系后，应满足：

- 同一物体的左右观测中心应接近
- 同一物体的左右观测点云应能较好重合

这一步非常关键，因为它能够避免：

- 左图看起来对，右图明显不对
- 左相机位姿和右相机位姿分别自洽，但彼此不一致

#### 第五步：联合优化得到最终相机位姿

最终不应分别求出左右相机位姿，而应通过统一目标函数联合优化：

- 左相机位姿
- 右相机位姿
- 必要时包括物体中心和平移

优化目标建议至少包括：

- 机械臂关键点投影误差
- 苹果和梨中心点投影误差
- 左右相机下物体轮廓误差
- 左右局部点云变换到世界系后的重合误差
- 与当前初值的偏离正则项

### 5.3 结论

相机位姿的最终答案不应是“通过某一个模块单独算出来的数值”，而应是：

- 在统一世界系下
- 同时满足背景、机械臂、双目物体和轮廓投影约束
- 并通过联合优化得到的场景级最优解

这才是能真正保证 MuJoCo 相机视角与真实图像一致的方式。


## 6. 物体位姿动态对齐方案

### 6.1 总体原则

物体位姿必须统一表示为完整的 6D pose：

- 平移 `T`
- 旋转 `R`

不能再延续“位置一套逻辑、旋转另一套逻辑”的结构。

对每个物体，都应维护：

- `T_world_object`

也就是物体相对于统一世界系的完整位姿。

### 6.2 物体位姿的推荐求解流程

#### 第一步：得到物体平移初值

继续利用当前 `pointcloud_v2.pos()` 的结果作为位置初值。

原因是：

- 现有 `pos()` 已经在当前项目中验证过相对位置关系可用
- 它仍然适合做初始化

但必须明确：

- `pos()` 的输出不应再直接作为最终物体位置真值
- 它只是物体世界平移的初值

#### 第二步：得到物体旋转初值

继续利用当前 `object_pose_runtime.py` 的方法作为旋转初值：

- 读取左相机原始点云
- 计算观测点云 PCA 主轴
- 读取 STL / mesh 本体
- 计算 mesh 的主轴
- 枚举有限个离散姿态候选
- 用 mask 投影评分挑选更优候选

这部分当前设计已经比较合理，应继续保留为姿态初始化器。

#### 第三步：引入双目共同验证

当前姿态估计主要由左图驱动，这是不够的。

本方案建议对每个物体增加双目联合验证：

- 将当前物体 mesh 以候选姿态投影到左图
- 计算与左图 mask 的一致性
- 同时将同一姿态投影到右图
- 计算与右图 mask 的一致性

最终每个候选姿态的得分应来自左右视角的联合评分，而不是只看左图。

#### 第四步：联合优化完整 6D pose

对每个物体，不只优化旋转，还应在当前相机外参下同时优化：

- `x`
- `y`
- `z`
- `roll`
- `pitch`
- `yaw`

优化目标建议包括：

- 左图物体轮廓与真实 mask 的 IoU
- 右图物体轮廓与真实 mask 的 IoU
- 左右图中物体中心投影误差
- 左右局部点云在当前 `T_world_object` 下与 mesh 的一致性
- 物体底部与桌面接触关系
- 物体不能悬空、穿桌、穿模

### 6.3 为什么必须同时优化平移和旋转

因为在现实图像中，物体的视觉表现并不是只由姿态决定，也不是只由位置决定，而是由完整 6D pose 决定。

如果只优化旋转：

- 轮廓可能方向对了，但遮挡关系仍不对
- 物体在 MuJoCo 中可能会漂浮或相对桌面错位

如果只优化平移：

- 物体中心位置看起来对，但朝向明显错误

因此，后续真正要动态对应现实场景的，不是单独的 `pos` 或单独的 `quat`，而是完整的 `T_world_object`。


## 7. 相机与物体的统一闭环

为了让仿真环境与现实场景保持一致，必须把相机与物体放进同一个闭环中。

统一闭环的核心思想是：

- 相机位姿影响物体投影结果
- 物体位姿影响相机对齐的优化结果
- 两者应在一个统一目标函数下共同成立

因此，正确的 real2sim 场景状态不应由以下方式得到：

- 相机位姿单独算
- 物体位置单独算
- 物体旋转单独算
- 最后把三者拼起来

而应由以下方式得到：

- 先构造统一世界系
- 再给出相机和物体的初值
- 然后联合优化相机和物体状态
- 最终输出统一场景状态

这套统一场景状态建议在概念上包含：

- 左右相机位姿
- 机械臂基座位姿
- 苹果完整位姿
- 梨完整位姿
- 桌面平面参数
- 各类重投影与轮廓误差指标


## 8. 如何判断仿真场景已经与现实场景一致

仅凭“肉眼感觉差不多”不够，需要定义一致性指标。

本方案建议至少检查以下四类指标。

### 8.1 关键点投影一致性

在左右图中检查以下点的投影误差：

- 机械臂基座点
- 机械臂主轴参考点
- 苹果中心
- 梨中心

这些点在仿真渲染和真实图像中的位置应足够接近。

### 8.2 轮廓一致性

将当前物体 mesh 渲染到左右图像平面，比较：

- 渲染轮廓
- 真实 mask

重点看：

- IoU
- 轮廓中心偏差
- 主轴方向偏差

### 8.3 深度 / 平面一致性

比较仿真和现实中的桌面 / 背景平面：

- 平面法向是否一致
- 平面高度是否一致
- 物体是否正确落在桌面上

### 8.4 几何合理性

最终场景必须满足：

- 物体不穿透桌面
- 物体不明显漂浮
- 左右相机渲染中的遮挡关系与真实图一致
- 机械臂、桌面、物体的空间关系合理


## 9. 与当前仓库的对应关系

这套方案与当前仓库并不冲突，而是对当前模块的职责做出重新组织。

### 9.1 `pointcloud_v2.py`

未来应继续承担：

- 左右局部点云生成
- 机械臂参考点提取
- 苹果、梨相对位置初值生成

但它输出的 `pos()` 更适合作为初始化，不应直接当最终物体世界位置真值。

### 9.2 `calibrate_runtime_pose_from_clouds.py`

未来应继续承担：

- 背景平面提取
- 世界系初始化
- 左右相机位姿初始化

但当前 `runtime_pose_calibration.json` 更适合作为初值和中间产物，不应直接视为最终 scene 真值。

### 9.3 `object_pose_runtime.py`

未来应继续承担：

- 原始点云主轴估计
- 物体 mesh 主轴估计
- 姿态候选生成
- 单物体局部打分

但它更适合作为物体姿态初始化器，而不是最终完整物体位姿的唯一来源。

### 9.4 `zhenghe2_buquan.py` 与 `dong2.py`

未来仍可保持：

- `zhenghe2_buquan.py` 作为主入口
- `dong2.py` 作为最终写入 MuJoCo scene 的模块

但它们应消费统一的场景状态，而不是分别接收多条独立链路的结果。


## 10. 最终建议

如果目标是“确保在仿真环境中相机看到的场景与现实相机拍摄的场景保持一致”，那么未来真正要建立的不是单一的相机标定模块，也不是单一的物体姿态模块，而是：

- 一个统一世界系
- 一个统一 scene state
- 一个统一的场景级联合优化过程

在这个框架下：

- 相机位姿由背景平面、机械臂结构、双目物体观测共同确定
- 物体位姿由双目点云、双目 mask、桌面接触关系和 mesh 模型共同确定
- 最终 MuJoCo 场景不再是“多个模块结果的拼装”，而是“同一个观测闭环下的最优一致解”

这才是当前项目从“可运行”走向“高一致性 real2sim”的关键升级方向。
