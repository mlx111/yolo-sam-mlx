# Grounded-SAM-2 误识别修复说明

## 问题现象

当前抓取流程中，输入 `pear` 时，有时会先识别到 `apple`。  
当苹果被抓走以后，再输入 `pear`，系统又能识别到真正的梨。

这说明问题不是 `pear` 完全检测不到，而是同一画面里同时存在多个相似目标时，当前识别链路把错误目标排到了第一位。

## 根因分析

原来的流程是：

1. 用户输入目标类别，比如 `pear`
2. `Grounding DINO` 用文本提示词去找目标框
3. 直接取第一个候选框
4. `SAM2` 对这个框做分割
5. 后续点云和抓取都基于这个 mask

这个流程的问题在于：

- `Grounding DINO` 属于开放词汇检测，不是专门做 `apple` 和 `pear` 细粒度区分的分类器
- 在桌面抓取场景里，苹果和梨外形接近时，模型可能把更显眼的苹果也当成和 `pear` 足够相关
- 原实现直接取第一个候选框，没有做二次判别
- 所以一旦苹果排在第一位，后面就会错误地把苹果送进 `SAM2`
- 苹果被抓走以后，梨变成场景里最显眼的目标，因此又会“恢复正常”

## 修复目标

修复的目标不是单纯“换一个模型”，而是：

- 保留现有的文本输入方式，仍然输入 `pear` / `apple` / `bowl`
- 保留 `Grounded-SAM-2` 这条主链路
- 不再盲信第一个检测框
- 在进入 `SAM2` 之前，先对候选目标做重排
- 当结果不可靠时，直接拒绝抓取，而不是误抓

## 修复后的流程

修复后的识别流程变成：

1. 用户输入目标类别，比如 `pear`
2. 系统先把输入标准化为小写形式
3. 如果目标属于当前固定场景类别集合，就同时查询多个相关类别
   - 当前集合默认是 `apple` / `pear` / `bowl`
4. `Grounding DINO` 分别对这些类别生成候选框
5. 系统只把 `pear` 相关候选作为主候选集
6. 对每个 `pear` 候选，检查它和其他类别候选的重叠关系
7. 如果某个 `pear` 候选与 `apple` 候选高度重叠，则给它施加惩罚
8. 根据最终得分对所有 `pear` 候选重新排序
9. 只有最终得分足够高、并且和第二名差距足够大时，才把该框送入 `SAM2`
10. `SAM2` 只负责对最终选中的框做精细分割

这样做以后，系统不再是“谁排第一就抓谁”，而是“先判断哪个候选更像真正的目标，再决定是否抓取”。

## 这次具体是怎么做到的

### 1. 增加受限场景类别集合

在 `Grounded-SAM-2/cv_proc.py` 中增加了固定场景类别集合：

- `apple`
- `pear`
- `bowl`

这样当输入 `pear` 时，系统不会只查询 `pear`，而是会同时观察当前场景里几个已知易混淆目标的响应。

### 2. 增加目标类别标准化

输入类别会先进行标准化处理：

- 去空格
- 转小写
- 去掉末尾句点

这样可以减少因为大小写或格式差异导致的开放词汇匹配不稳定。

### 3. 从“单次检测”改成“多类别候选收集”

原来是：

- 只用用户输入的类别跑一次检测
- 直接拿第一批结果

现在是：

- 对目标类别相关的多个类别分别跑检测
- 收集所有候选框
- 保留每个候选框的类别、置信度和位置

### 4. 引入候选重排机制

这是这次修复的核心。

系统会先取目标类别的候选框，例如输入 `pear` 时，只把 `pear` 检测结果作为待选目标。  
然后再看这些候选框是否和 `apple` 等混淆类候选框重叠。

如果某个 `pear` 候选和一个高置信度的 `apple` 框高度重叠，那么它很可能只是“把苹果误当成了梨”，这时就会被扣分。

最终分数由几部分组成：

- 目标类别本身的置信度
- 混淆类别重叠惩罚
- 其他类别重叠惩罚
- 中心区域轻微加分
- 较合理目标面积的轻微加分

整体思想是：

- 优先选择“像目标类，并且不像混淆类”的候选
- 而不是简单选择“原始置信度最高”的候选

### 5. 增加不确定性拒绝机制

为了避免误抓，增加了两层保护：

- 如果最佳候选的最终得分过低，直接判定不可靠
- 如果第一名和第二名得分太接近，说明区分不明显，也直接判定不可靠

这时系统会返回空 mask，主流程会跳过本轮抓取。

这样的好处是：

- 宁可这轮不抓
- 也不要抓错苹果

### 6. 让 `SAM2` 只做分割，不做语义决策

修复前：

- `SAM2` 实际上是在错误框的基础上输出错误 mask

修复后：

- 先把目标框选对
- 再让 `SAM2` 分割这个框

也就是说，这次修复没有把问题推给 `SAM2`，而是把语义选择问题放回检测和候选排序阶段解决。

### 7. 增加主循环的空结果保护

主流程里增加了更稳的判断：

- 如果返回值是 `None`
- 或者返回的是全零 mask

就直接跳过本轮，而不是继续做后面的点云和抓取。

这样可以避免“不确定识别”继续传到后续模块，导致抓取阶段出现错误行为。

## 为什么这个方案能解决你的现象

你的现象本质上是：

- 苹果和梨同时存在时
- `pear` 查询得到的第一候选不一定真的是梨
- 苹果被移走以后，梨才变成最强候选

修复后，系统不再只看 `pear` 检测结果本身，而是会同时参考：

- 这个候选像不像 `pear`
- 这个候选是不是同时也很像 `apple`
- 它和 `apple` 的候选框是否高度重叠

因此，当某个 `pear` 候选其实更像苹果时，它会被惩罚，真正的梨候选更容易排到前面。  
如果两者仍然分不清，系统会拒绝抓取，而不是误抓。

## 当前默认参数

当前加入了几项默认阈值：

- `TARGET_TOP_K = 5`
- `OVERLAP_IOU_THRESHOLD = 0.25`
- `FINAL_SCORE_THRESHOLD = 0.08`
- `SELECTION_MARGIN_THRESHOLD = 0.05`

含义如下：

- 最多保留前 5 个目标候选做重排
- 两个框重叠度超过 0.25 时，认为它们可能在指向同一物体
- 最佳候选最终得分太低时，不执行
- 第一名和第二名差距太小时，也不执行

这些参数是第一版工程默认值，后续可以结合你的实际场景继续调。

## 调试时可以看什么

现在运行时会输出一份候选排序日志，里面会显示：

- 每个候选来自哪个查询类别
- 原始置信度
- 最终得分
- 与混淆类的冲突惩罚
- 与其他类的惩罚

同时也会把候选框和最终选中的框画到：

- `outputs/groundingdino_*.jpg`

调试时可以重点看：

- 输入 `pear` 时，苹果框是不是被标出来了
- 苹果框和梨框是否有明显重叠
- 梨候选的 `final_score` 是否高于苹果误匹配带来的干扰
- 如果系统拒绝抓取，是因为分数太低，还是因为前两名太接近

## 本次修改的工程意义

这次修改不是简单“提高检测阈值”，而是把识别流程从：

- 开放词汇检测直接决定抓取对象

改成了：

- 开放词汇检测先提名候选
- 工程逻辑再做目标选择和风险控制

这样更适合机械臂抓取场景，因为抓取系统更重要的是：

- 选对目标
- 不误抓
- 不确定时安全退出

而不是单纯让检测器尽量多出结果。

## 后续如果还想继续增强

如果后面还需要继续提升 `apple` / `pear` 区分稳定性，可以继续做这几个方向：

1. 加入深度先验
   - 结合深度图判断哪个候选更接近工作区中心或抓取区域

2. 加入历史状态
   - 如果上一轮刚抓走苹果，下一轮可以降低苹果相关候选的优先级

3. 换检测器但保留同样的重排逻辑
   - 比如后面换成 `YOLO-World`
   - 仍然保留“多候选重排 + 不确定拒绝”的策略

4. 为固定目标增加局部分类器
   - 在检测框裁剪图上增加一个小型 `apple/pear` 二分类器
   - 用于进一步打破开放词汇检测的歧义

## 涉及文件

本次核心修改涉及：

- `Grounded-SAM-2/cv_proc.py`
- `main_yoloWorld_sam_completion1.py`

其中：

- `cv_proc.py` 负责目标候选收集、重排和最终框选择
- `main_yoloWorld_sam_completion1.py` 负责在拿不到可靠 mask 时安全跳过本轮
